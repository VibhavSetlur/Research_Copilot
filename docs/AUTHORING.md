# AI-Driven Capability Authoring

In Research OS, capabilities and tools are not statically authored by human engineers. Instead, **AI agents dynamically author and extend** the system's capabilities at runtime via MCP protocols.

## Dynamic ToolRegistry Extension

When a `CriticAgent` or `SupervisorAgent` determines that a necessary tool is missing, it constructs a new tool definition and submits it directly to the `ToolRegistry` via an MCP endpoint. 

Example payload generated by the AI:
```json
{
  "tool_name": "run_regression",
  "capabilities": ["statistical_analysis", "linear_regression"],
  "cost": "medium",
  "risk": "low",
  "failure_modes": ["missing_dependency", "data_shape_error"],
  "preferred_when": ["hypothesis_testing", "correlational_analysis"],
  "inputSchema": {
    "type": "object",
    "properties": {
      "dataset_path": {"type": "string"},
      "target_variable": {"type": "string"}
    },
    "required": ["dataset_path", "target_variable"]
  }
}
```

## Capability Gating
Once an agent authors a tool, the `ToolCapabilityCheck` component within `validation/safety.py` governs its execution. High-risk tools (e.g., executing code or accessing external servers) require an elevated confidence score (>0.85) from the cognitive subsystem before the OS permits execution.

This paradigm ensures the OS organically grows its skillset based strictly on the semantic demands of the research execution DAG.
