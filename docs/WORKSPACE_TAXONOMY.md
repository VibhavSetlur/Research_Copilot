# Workspace Taxonomy

Every Research OS project follows a strict, reproducible directory structure. This document explains every folder and file.

---

## Top-Level Structure

```
<user-project>/
├── AGENTS.md                # AI agent instructions
├── README.md                # Auto-generated project overview
├── .cursor/rules/           # Cursor-specific rules
├── .os_state/               # INTERNAL — OS state (DO NOT EDIT)
├── docs/                    # Research documentation
├── inputs/                  # IMMUTABLE — original data (read-only by tools)
├── workspace/               # Active research area
├── synthesis/               # Final paper outputs
└── environment/             # Reproducible environments
```

---

## `inputs/` — Immutable Source Data

Write-protected at the tool level. Tools will raise `WriteProtectedError` if they attempt to modify anything under `inputs/`.

```
inputs/
├── raw_data/            # Source data files (CSV, Parquet, JSON, etc.)
│   └── survey.csv
├── literature/          # PDF papers (keep original filenames)
│   ├── smith2024.pdf
│   └── jones2023.pdf
├── context/             # Supporting context files
├── literature_index.yaml   # Auto-generated: maps filename → citation_key, sha256, verified
└── intake.md               # Auto-generated: research question, domain, SHA-256 table of all input files
```

### `intake.md`

Generated by `regenerate_intake()` (called during `init` and every `mem.regenerate.intake`). Contains:
- Project name, domain, depth
- Research question
- Keywords
- Table of every input file with SHA-256 hash and size

### `literature_index.yaml`

Generated by `update_literature_index()`. Maps each PDF filename to:
- `citation_key`: auto-derived from stem (e.g., `smith2024`)
- `sha256`: content hash for integrity verification
- `verified`: boolean flag for citation verifier

## `workspace/` — Active Experimentation

The core research area. All analysis happens here.

```
workspace/
├── 01_experiment_baseline/  # Numbered experiment directories
│   ├── README.md            # Goal, methods, outcomes, next-step decision
│   ├── conclusions.md       # Key findings and next steps
│   ├── data/                # Experiment-specific derived data
│   ├── scripts/             # Versioned analysis scripts (01_load_v1.py, ...)
│   ├── outputs/
│   │   ├── figures/         # Publication-ready figures
│   │   ├── reports/         # Analysis reports
│   │   ├── tables/          # Summary tables
│   │   └── dashboards/      # Interactive dashboards
│   └── environment/         # Pinned dependencies
├── 02_data_preparation/     # Next experiment (created by sys.path.create)
│   └── ...
├── analysis.md              # Chronological log + Mermaid workflow diagram
├── methods.md               # Append-only method registry
├── citations.md             # Running bibliography
├── workflow.mermaid         # Auto-updated state diagram
├── workflow.png             # Rendered diagram (if mmdc installed)
├── logs/
│   ├── searches.log         # Every web search logged (JSON lines)
│   ├── state_changes.log    # Before/after diffs of state ledger
│   ├── notifications.log    # Researcher notifications
│   ├── data_inventory.json  # Auto-profiled data inventory
│   └── ...                  # Per-step execution logs
└── .os_state/               # Symlink to root .os_state/
```

### Numbered Experiment Folders (`01_`, `02_`, ...)

Created by `sys.path.create`. Each folder is a self-contained step.

**Path-based chronological system:** Experiments run consecutively, not in parallel branches. Each gets an auto-incrementing number (`01_experiment_baseline/`, `02_data_preparation/`, `03_hypothesis_testing/`). If a path reaches a dead end, use `sys.path.abandon` to rename it with a `__DEAD_END__` or `__ABANDONED__` suffix — files are preserved, not deleted. The `analysis.md` log records every path change and reroute.

**`README.md`** contains:
- **Goal**: What this experiment aims to discover
- **Input Data**: Files and sources used
- **Methods Used**: Statistical tests, transforms, models
- **Expected Output**: Predicted results before execution
- **Actual Output**: Results after execution
- **Next-Step Decision**: proceed / branch / dead-end

**`conclusions.md`** contains:
- Summary of findings
- Key metrics and effect sizes
- Next steps

### `analysis.md`

The "state machine" log. Updated by `sys.analysis.log` after every significant step.

Contains:
- **Mermaid workflow diagram**: auto-updated with status-colored nodes (green=complete, yellow=running, grey=planned, red=failed, dashed red=dead-end)
- **Chronological log entries**: `[timestamp] step_name: description`

### `methods.md`

Append-only. Every tool that runs a statistical test, data transformation, or literature retrieval must append a structured entry via `mem.methods.append`:
```markdown
## 2026-05-22 14:03:00
- **Method**: Welch t-test
  - **Parameters**: equal_var=False, alpha=0.05
  - **Tool**: tool.search.semantic_scholar / manual
  - **Citation**: student1908
```

### `citations.md`

Running bibliography. Generated from `literature_index.yaml` by `mem.citations.generate`. Each entry has a `verified` flag:
```markdown
### `smith2024`
  - **File**: smith2024.pdf
  - **SHA-256**: `a1b2c3d4e5f6...`
  - **Status**: ⏳ Pending verification
```

## `synthesis/` — Final Outputs

Populated only when the user triggers `sys.synthesize` or `sys.scaffold.synthesis`.

```
synthesis/
├── abstract.md          # 250-word structured abstract template
├── paper.tex            # LaTeX skeleton
├── references.bib       # Bibliography from citations.md
├── supplementary/       # Supplementary materials
└── paper.pdf            # Compiled PDF (after tool.latex.compile)
```

Created during `init`. Contains structured research documentation:

```
docs/
├── research_question.md     # Main question and sub-questions
├── hypotheses.md            # H1, H2, ... with test plans
└── glossary.md              # Term definitions
```

## `.os_state/` — Internal State (DO NOT TOUCH)

Managed entirely by Research OS. Manual edits may corrupt the state machine.

```
.os_state/
├── state_ledger.yaml      # Primary source of truth (YAML)
├── manifest.json          # Workspace manifest with file checksums
├── checkpoints/           # Workspace snapshots (created by sys.checkpoint)
│   ├── before_model_training/
│   │   └── checkpoint_manifest.json
│   └── ...
└── cache/                 # API response cache
```

### `state_ledger.yaml`

The single source of truth. Contains:
- `project_id`, `project_name`
- `current_path`, `step`, `pipeline_stage`
- `paths`: all experiment paths with status, hypothesis, creation time
- `checkpoint_history`: ordered list of snapshots
- `updated_at`: ISO timestamp of last change

Updated atomically (temp file + rename) after every tool execution. Before/after diffs logged to `workspace/logs/state_changes.log`.

---

## Conventions Summary

| Rule | Enforcement |
|------|-------------|
| `inputs/` is immutable | `WriteProtectedError` at tool level |
| Experiments are numbered `01_`, `02_`, ... | `sys.path.create` auto-numbers |
| Every experiment has a `README.md` | Created by `create_numbered_experiment()` |
| `methods.md` is append-only | `mem.methods.append` appends, never overwrites |
| `analysis.md` contains Mermaid diagram | `sys.analysis.log` updates diagram + log |
| State changes are logged | `save_state()` writes diff to `state_changes.log` |
| Files have SHA-256 checksums | Every tool response includes checksums |
